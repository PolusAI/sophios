# User Guide

Sophios lets you describe a workflow in a small YAML DSL, infer most of the wiring between steps, compile the result to CWL, and optionally run it locally.

## Main Features / Design Overview

See [overview](overview.md).

## Auto-discovery

Sophios discovers CWL tools and `.wic` workflows from the JSON config file entries `search_paths_cwl` and `search_paths_wic`.

If you do not pass `--config_file`, Sophios uses `~/wic/global_config.json`. If that file does not exist yet, the CLI creates it automatically on first run. The same first-run setup also copies the bundled adapters and example workflows into:

- `~/wic/cwl_adapters`
- `~/wic/examples`

You can trigger that setup explicitly with:

```bash
sophios --generate_config
```

The generated config starts from the defaults in the codebase and points at those copied directories. For example:

```json
{
  "search_paths_cwl": {
    "global": [
      "/path/to/your/home/wic/cwl_adapters"
    ],
    "gpu": []
  },
  "search_paths_wic": {
    "global": [
      "/path/to/your/home/wic/examples"
    ]
  }
}
```

You can add more directories at any time. A few practical details matter:

- Discovery is recursive, so subdirectories are searched too.
- CWL tools are keyed by filename stem within a namespace.
- `.wic` workflows are also keyed by filename stem within a namespace.
- If two files in the same namespace share a stem, the later discovery overwrites the earlier one in memory.
- Relative paths in the config are resolved from your current working directory when Sophios reads the config, so absolute paths are safer.

If you need duplicate names on purpose, use [namespaces](advanced.md#namespaces).

## Running Workflows

The smallest runnable workflow is [docs/tutorials/helloworld.wic](tutorials/helloworld.wic):

```yaml
steps:
- echo:
    in:
      message: !ii Hello World
```

Run it locally with:

```bash
sophios --yaml docs/tutorials/helloworld.wic --run_local --copy_output_files --quiet
```

The most common CLI modes are:

- `sophios --yaml <workflow>.wic --generate_cwl_workflow` compiles to CWL and writes artifacts to `autogenerated/`.
- `sophios --yaml <workflow>.wic --run_local` compiles and runs locally with the configured CWL runner.
- `sophios --yaml <workflow>.wic --generate_run_script` writes `run.sh` instead of executing it.
- `sophios --generate_schemas` regenerates `autogenerated/schemas/wic.json` and related schema artifacts.
- `sophios --generate_config` writes the default config if it does not already exist.

Useful flags from the current CLI:

- `--graphviz` writes Graphviz sources and rendered diagrams into `autogenerated/` when `dot` is installed.
- `--inputs_file <file>` merges extra job inputs into the generated CWL inputs document.
- `--copy_output_files` mirrors primary outputs into `outdir/`.
- `--cwl_runner toil-cwl-runner` switches the local runner from `cwltool`.
- `--container_engine podman` runs containers with Podman instead of Docker.

## Generated Files

During compilation and local execution, Sophios writes several useful artifacts:

- `autogenerated/<workflow>.cwl` is the compiled root CWL workflow.
- `autogenerated/<workflow>_inputs.yml` is the generated root input object.
- `autogenerated/*_tree_raw.wic`, `*_tree_merged.wic`, and `*_tree_python_script.wic` are intermediate workflow snapshots that are useful for debugging.
- `autogenerated/schemas/` contains generated JSON schemas for editor validation.
- `cachedir*` stores CWL runner caches and intermediate results.
- `provenance/` contains CWL provenance data unless you pass `--no_provenance`.
- `outdir/` receives copied primary outputs when you pass `--copy_output_files`.
- `error_<workflow>.txt` captures a technical traceback when compilation or execution fails.

## Edges

In many workflow systems, an edge simply means "step B depends on step A". CWL is more precise: edges connect individual outputs to individual inputs. That extra structure is what makes Sophios inference possible.

## Edge Inference Algorithm

Sophios only connects an input in the current step to outputs that already exist from earlier steps.

At a high level, the compiler walks backward through previous steps and looks for outputs whose `type` and `format` match the current input. Because many workflows are roughly linear, the search prefers the most recent compatible output. If there is exactly one compatible match, that edge is inferred automatically. If there are multiple matches, Sophios chooses the first compatible one it finds.

For technical reasons edge inference is not unique, so ***`users should always check that edge inference actually produces the intended DAG`***.

If type and format matching alone are too coarse, you can enable `--inference_use_naming_conventions`. The default renaming rules live in the config file under `renaming_conventions`. See [advanced features](advanced.md#edge-inference-configuration) for the details.

## Explicit Edges

If inference does not pick the edge you want, you can always wire it explicitly with `!&` and `!*`.

- Use `!& name` on an output to define an anchor.
- Use `!* name` on a later input to consume that anchor.

Example:

```yaml
steps:
- touch:
    in:
      filename: !ii empty.txt
    out:
    - file: !& created_file
- cat:
    in:
      file: !* created_file
```

This notation is intentionally similar to YAML anchors and aliases, but it is specific to workflow edges.

## Inline Inputs

If you already know a literal input value, you can provide it inline with `!ii` and skip both inference and explicit edges.

```yaml
steps:
- echo:
    in:
      message: !ii Hello World
```

That is one of the biggest user-facing differences between Sophios and raw CWL. In CWL, job inputs normally live in a separate file. In Sophios, inline values are accepted in the workflow source and then extracted into the generated CWL inputs document during compilation.

If you need to treat the input YAML more like raw CWL, you can use `--allow_raw_cwl`.

## Troubleshooting

When a run fails:

- Scroll to the first error message. Later errors are often just fallout.
- Re-run without `--quiet` if you need the full CWL runner output.
- Inspect `error_<workflow>.txt` if the failure happened during compilation or runner startup.
- Delete `autogenerated/`, `cachedir*`, `outdir/`, and `provenance/` if you want to force a clean rebuild.

## Python API (experimental)

Sophios also exposes a Python API for building the same workflows programmatically. The examples in [examples/scripts](https://github.com/PolusAI/sophios/tree/main/examples/scripts) are the best starting point:

- `helloworld_pyapi.py` shows the smallest `Step` + `Workflow` example.
- `multistep1_pyapi.py` shows explicit step-to-step wiring in Python.
- `scatter_pyapi.py` shows `scatter` and `scatterMethod`.
- `when_pyapi.py` shows conditional execution with `when`.

A minimal example looks like this:

```python
from sophios.apis.python.api import Step, Workflow


def workflow() -> Workflow:
    echo = Step(clt_path="../../cwl_adapters/echo.cwl")
    echo.message = "hello world"
    return Workflow([echo], "helloworld_pyapi_py")


if __name__ == "__main__":
    workflow().run()
```

For the Python API, it helps to keep three methods in mind:

- `Workflow.yaml` returns the in-memory workflow DSL document.
- `Workflow.get_cwl_workflow()` returns the compiled CWL plus generated inputs in memory.
- `Workflow.run()` compiles and executes the workflow locally.
